# 纯本地色卡识别方案复杂度复盘

本文记录“完全本地化识别”方案在本项目中的实际复杂度、已遇到的问题和最终取舍。结论先行：纯本地方案不是不能做，而是工程成本、维护成本、标注成本和异常兜底成本都明显高于预期；在当前业务场景下，云端视觉模型识别更适合作为主路径。

## 背景

项目早期目标是尽量做到彻底本地化：软件安装后不依赖网络，直接在本机完成色卡图片识别、分组、色号提取和 Word 模板生成。

为实现这个目标，方案逐步演变为：

1. 用标注平台手动给图片画框，标出 `name_area` 和 `code_area`。
2. 导出 YOLO 训练数据。
3. 分别训练竖版和横版 YOLO 模型。
4. 软件本地加载 YOLO 模型，检测组名区域和色号区域。
5. 将检测框裁剪出来交给 RapidOCR。
6. 对 OCR 结果做规则清洗、去重、缺号判断、顺序修正。
7. 最后再进入表格确认和 Word 生成。

这个流程理论上完整，但实际每一层都有独立的不确定性。

## 复杂度来源

### 1. 横版和竖版不是一个问题

竖版和横版色卡的布局差异很大。

竖版中，色号区域可能是两列，也可能是三列；如果 YOLO 把三列识别成两列，程序很难仅凭结果知道漏了哪一列。横版中，色号通常是一整条横向排列，但数字之间的空白、背景线条和装饰元素会影响 OCR。

这意味着模型、标注策略、裁剪策略和后处理规则都要按版式分别处理，不能简单认为“检测框 + OCR”就是通用解法。

### 2. 标注质量直接决定模型边界

YOLO 的检测框依赖人工标注。框标得偏大，OCR 会吃到背景线条、边框、纹理；框标得偏小，数字可能被截掉；不同批次图片如果边界习惯不一致，模型输出也会不稳定。

这导致问题排查经常要回到原始标注：到底是模型错、标注错、裁剪错，还是 OCR 错。每类问题的修复方式都不同。

### 3. OCR 对输入非常敏感

RapidOCR 在本项目中遇到过几个典型问题：

- 数字 `9` 被识别成 `6`。
- 数字之间有空格，但被连读成一串。
- 框内只多了一点横线或背景纹理，识别结果就发生变化。
- 同一张裁剪图，用 `PIL.Image`、文件路径、`np.ndarray` 输入时，结果可能不一致。
- EXIF 旋转标记会导致图像显示方向和实际像素方向不一致。

这些问题说明本地 OCR 并不是一个稳定的黑盒。为了提升准确率，需要继续做图像预处理、输入格式控制、裁剪边界调整、旋转处理和异常规则。

相关调查见：

- `docs/rapidocr-input-channel-investigation.md`
- `docs/ocr-heuristic-risk-analysis.md`

### 4. 后处理规则会不断膨胀

为弥补 OCR 的不稳定，本地方案需要不断增加规则，例如：

- 重复框去重。
- 空结果丢弃。
- 缺号判断。
- `1,2,3,4,5,6,7,8,6` 中第二个 `6` 可能应修正为 `9`。
- 连读数字尝试拆分。
- 裁剪失败后扩大或收缩框重试。
- 低置信度 YOLO 框是否采信。

这些规则短期可以解决单个样本，但长期会造成维护风险：一个规则可能修好 A 图，同时破坏 B 图。尤其当色号并不总是连续数字，还可能出现 `A1`、`A2` 这类编号时，纯启发式拆分会变得非常危险。

### 5. “没识别出来”不等于同一种错误

本地方案中，同样表现为“缺号”或“没识别到数字区域”，背后可能是完全不同的原因：

- 图片本身确实缺号。
- YOLO 没有圈到区域。
- YOLO 圈到了，但框偏大或偏小。
- OCR 没读出来。
- OCR 读出来了，但没有通过有效色号判断。
- 后处理把结果过滤掉了。

因此排查时不能只看最终表格，还要看原图、YOLO 框、裁剪图、OCR 原始文本、过滤原因和最终清洗结果。这要求软件必须增加大量可观测性，否则很难定位问题。

### 6. 本地并发能力有限

本地 YOLO、OCR、图像预处理都占用 CPU、内存和模型运行资源。提高并发数不一定能线性加速，反而可能导致：

- 内存占用明显增加。
- 多模型或多 OCR 实例初始化变慢。
- CPU/GPU 抢占导致整体变慢。
- 打包体积继续变大。

这也是纯本地桌面软件安装包很容易变到 1GB 左右的重要原因：为了离线推理，需要把模型运行时、深度学习依赖、OCR 依赖和相关动态库都带上。

## 已做过的工程投入

本项目为了推进本地方案，已经做过这些工作：

- 建立本地标注平台。
- 整理原始图片目录和标注导出目录。
- 导出 YOLO 数据集。
- 分别训练竖版和横版模型。
- 接入本地 YOLO 推理。
- 接入 RapidOCR。
- 处理 RGB/BGR 输入差异。
- 处理 EXIF 旋转。
- 增加横版、竖版不同识别流程。
- 增加 OCR 失败后的手动可编辑结果。
- 增加日志和识别统计。

这些投入证明本地方案可行，但也暴露出它不是低成本路径。

## 云端视觉模型方案的优势

当前推荐方案是使用云端视觉模型直接识别色卡左上角组名和色号列表，并返回结构化 JSON。

优势主要有：

- 不再依赖本地 OCR 对裁剪边界的敏感性。
- 能整体理解版式，天然适合两列、三列、横排、含 `A1/A2` 等复杂编号。
- 不需要频繁为新样本重新训练模型。
- 可以使用并发请求提升速度。
- token 成本可直接统计，成本透明。
- 软件安装包仍可保留本地兜底，但主路径不再被本地推理复杂度拖住。

在 `qwen3.6-flash` 这类模型上，实测单张图片成本通常约为 `0.3 分`人民币量级。开启 10 并发后，批量处理速度也明显优于本地串行 OCR 路径。具体成本以模型价格和接口返回 token 为准。

## 当前取舍

保留本地能力作为兜底是有价值的，但不建议继续把纯本地识别作为主投入方向。

当前更合理的策略是：

1. 主路径使用云端视觉模型识别。
2. 横版默认整图云端识别，必要时可开启 YOLO 裁剪以降低 token。
3. 竖版直接整图云端识别，避免两列/三列检测不确定性。
4. 软件记录 token、费用、实际耗时、API 耗时合计、并发倍率和失败原因。
5. 本地 YOLO/OCR 保留为兜底能力，而不是继续堆复杂规则作为主路径。

## 经验教训

本项目的核心教训是：识别类项目不能只看“能不能跑通一张图”，必须先评估异常样本、维护成本、标注成本、并发能力、打包体积和可观测性。

如果一开始先做小规模技术调研，对比本地 OCR、传统视觉分割、YOLO 裁剪和云端视觉模型的准确率、速度与成本，后续可以少走很多弯路。

纯本地化听起来简单，实际是把所有模型能力、异常处理、算力压力和工程维护都压到客户端。对于当前色卡识别任务，云端视觉模型是更现实、更稳定、更便宜的主方案。
