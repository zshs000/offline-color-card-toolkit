# 启发式 OCR 补丁策略风险说明

## 核心结论

在样本数量不足、版式变化未被充分覆盖、识别逻辑主要依赖启发式规则的情况下，不能采用“遇到一个 bug 修一个 bug、遇到一个洞打一个补丁”的方式持续推进。

这种做法短期看可以让某一张图片通过，但长期会让规则越来越偏向少数样本，最终在其他正常样本上产生更大的偏差。问题不是某一个补丁写得不够仔细，而是这种修法本身会把局部经验错误地扩展成全局规则。

## 为什么补丁式修法会失控

### 1. 单个样本不代表整体规律

OCR 识别的失败原因可能很多，例如：

- 数字被反光干扰，识别成其他数字。
- 两个字符被合并成一个文本块。
- 某一行真实缺号。
- 竖向列被整体合并。
- 局部裁剪角度、光照、边缘阴影不同。

如果只根据一张失败图片添加规则，很容易把这张图的特殊情况误判为通用规律。

例如，某张图里 `27` 没被 OCR 识别出来，程序如果自动补齐 `27`，这张图可能看起来正确。但另一张图里本来就故意缺 `17`、`20`、`42`，同样的自动补齐规则就会把不存在的色号写进结果。

### 2. 启发式规则之间会互相干扰

启发式规则通常不是独立的。一个规则改的是“数字纠错”，另一个规则改的是“竖向排序”，第三个规则改的是“缺号补齐”。这些规则叠加后，实际效果可能不是简单相加，而是互相放大。

例如：

- 为了修 `蛋白绵羊` 漏 `27`，加入了连续编号补齐。
- 这个规则在 `狂飙风暴` 上把真实缺失的 `17`、`20`、`42` 当成 OCR 漏识别。
- 为了让连续序列更好看，又可能触发自动顺序修正。
- 在 `肤感荔枝` 上，原图左列顺序是 `1, 4, 3, 2, 5...`，程序却可能改成 `1, 2, 3, 4, 5...`。

结果就是：每个补丁都在解决一个局部问题，但组合起来会改变业务事实。

### 3. OCR 结果不能被当成“可自动推理的真相”

OCR 是概率输出，不是结构化数据源。它只能告诉我们“图上可能识别到了什么”，不能可靠证明“图上应该有什么”。

因此程序可以做：

- 坐标排序。
- 噪声过滤。
- 明确的字符归一化，例如 `O` 和 `0` 的局部纠正。
- 缺号提示。

但程序不应该轻易做：

- 凭空补充 OCR 没识别到的色号。
- 把真实识别到的编号改成连续编号。
- 假设所有色卡编号必然连续。
- 假设同一类版式永远使用同一套编号规则。

一旦程序开始“替用户推理图上应该有什么”，就会从识别工具变成自动改写工具，风险会明显上升。

## 当前项目中的具体表现

### `蛋白绵羊`

曾经出现 OCR 漏掉 `27` 的情况。为了让这张图通过，加入了自动补齐逻辑。

这类修法的问题是：它没有真正证明图上存在 `27`，只是根据连续编号假设补出来。

### `狂飙风暴`

原图里存在真实缺号，例如左列缺 `17`、`20`，右列缺 `42`。这些缺号是图片事实，不是 OCR 错误。

如果程序为了保持连续编号自动补齐，就会生成不存在的色号。

### `肤感荔枝`

原图左列顺序不是标准连续顺序，而是 `1, 4, 3, 2, 5...`。

如果程序为了“纠正顺序”强行改成 `1, 2, 3, 4, 5...`，就会破坏原图真实顺序。

这些例子说明：补丁式启发规则会从“修 OCR”滑向“改业务数据”。

## 建议原则

### 1. 识别结果必须尊重图片事实

程序只能写入识别到的内容，并按照图片中的空间位置排序。

如果中间缺号，应该提示“疑似缺少”，而不是自动补号。

### 2. 缺号只能提示，不能默认修复

推荐策略：

- 识别到：写入。
- 没识别到：不写入。
- 数字序列中断：弹出提示。
- 用户确认后再人工处理。

这样即使 OCR 不完美，也不会把错误数据静默写进 Word。

### 3. 自动纠错必须限定在低风险范围

可以保留的低风险规则：

- 去除明显广告文字、货名、规格等噪声。
- 按坐标进行左右列、上下顺序排序。
- 全角半角符号归一化。
- 极明确的字符归一化，但必须谨慎。

不应保留的高风险规则：

- 自动补齐缺失编号。
- 自动把非连续顺序改成连续顺序。
- 根据前后编号强行改写当前编号。

### 4. 后续优化应基于样本集，而不是单张图片

如果要继续提高准确率，应先建立样本集：

- 横版样本。
- 竖版样本。
- 连续编号样本。
- 故意缺号样本。
- 中文名、英文名、字母数字混合编号样本。
- 光照差、反光强、倾斜、裁剪不完整样本。

每次修改规则前，都要在这组样本上回归验证。只有在整体通过率提升且没有引入明显新错误时，规则才应该保留。

## 推荐调整方向

当前第一功能的 OCR 逻辑应调整为：

1. 保留版式裁剪和坐标排序。
2. 保留顶部/底部/右侧说明文字过滤。
3. 删除自动补齐缺号逻辑。
4. 删除按连续序列强行改写编号的逻辑。
5. 将缺号、低置信度、疑似顺序异常作为 warning 暴露给用户。
6. 在界面中允许用户查看识别结果并手动修正。

这个方案不会保证每张图都一次性完美识别，但它能保证一个更重要的底线：程序不会静默生成图片里不存在的数据。

## 结论

在 OCR 样本不足、版式复杂、业务允许真实缺号的场景下，补丁式启发规则会快速积累技术债，并把局部样本的特殊性误扩散为全局逻辑。

当前项目应该从“自动猜对”转向“识别事实 + 暴露风险 + 人工确认”。这比不断补洞更稳定，也更符合业务数据准确性的要求。
